# **תבניות וארכיטקטורת מערכת (System Patterns)**

## **1. חוקי יסוד (Hard Invariants)**
1. **הפרדה מוחלטת (Absolute Decoupling):** חל איסור מוחלט על הטמעת טקסט רפואי/לימודי כ-Hardcoded בתוך קוד רכיבי הממשק (UI). כל קומפוננטת UI חייבת להיות Dumb Component הניזונה מקבצי ה-JSON המתוייגים שבתיקיית `content_data/`. התוצרים הסופיים חייבים להיות נטולי HTML או CSS לחלוטין.
2. **חוק איסור ויקיפדיה:** חל איסור מוחלט לשאוב השראה, לחקות, או להתייחס לממשק הישן דמוי-ויקיפדיה. כל העיצוב החדש מחויב להיות Mobile-First, מבוסס מחוות גלילה (Swipe/Reels) ואנימציות.

## **2. אסטרטגיית ניהול תוכן קליני (Clinical Content Architecture)**

התוכן הרפואי אינו נכתב על ידי מתכנתים. הסוכן הקליני מייצר קבצי נתונים טהורים הנשמרים בתיקיית app_build/content_data/lessons/. כל יחידת לימוד מחויבת לעבור הערכת ראיות, ולהסביר את המנגנונים הפיזיקליים והפיזיולוגיים הרלוונטיים באופן מדויק ומבוסס מחקר. שיטות טיפול ספציפיות (כגון Halliwick, Jahara, Bad Ragaz) ייכללו רק כאשר הן מהוות חלק אינטגרלי מהעיקרון הנלמד, ולא כאזכור חובה בכל יחידה.  
**תבנית הנתונים (Strict JSON Schema):**

## 3. הגדרת בלוקים לימודיים וחוקי תוכן (Learning Blocks & Content Rules)

**חוקי תוכן אבסולוטיים (הוספו עקב דחיסת יתר):**
*   **חוק ההקשר (Contextual Foundation):** חובה לשאול "מהו הרקע ההכרחי להבנה?". לפני כניסה לנושא מעשי, יש לייצר רילסים תיאורטיים (`principle_card`) המסבירים את המערכת הפיזיולוגית/אנטומית כדי לבסס הבנה. *הערה חשובה: אין לדרוס או למחוק טקסטים קיימים וטובים, אלא רק להוסיף את ההקשר החסר לפניהם ולשמור עליהם ככל הניתן.*
*   **חוק איסור דחיסה (No Conceptual Compression):** רכיב ה-`flashcard_carousel` **לא ישמש להוראת נושאי ליבה** (למשל רשימת כל השרירים בקרוסלה אחת). קרוסלה תשמש *אך ורק* לציון דוגמאות קצרות. נושאי ליבה יופרדו לרצפים נפרדים (הסבר -> מקרה קליני -> קרוסלת תרגילים).
*   **העשרה ללא תלות באנימציה:** במקום להמציא `visual_trigger` לא קיימים, יש להעשיר את המסכים במידע טקסטואלי נעים, רכיבי מיקרו (כגון שאלות סקרנות "הידעת?"), ריווח ואייקונים קיימים.

על המערכת להשתמש בבלוקים מובנים כדי לייצר למידה דינמית:
1. **principle_card (כרטיס עקרון):** מסך בסיס עם אנימציה, טקסט תיאורטי (עד 40 מילים) ונקודת ציון קלינית.
2. **clinical_case_card (מקרה קליני):** יישום העיקרון על מטופל ספציפי ומה נכון לעשות.
3. **flashcard_carousel (קרוסלה):** רשימת מאפיינים שניתן לגלול ימינה/שמאלה במקום למטה.
4. **compare_card (מסך השוואה):** מסך מפוצל המראה הבדלים (למשל מים מול יבשה).
5. **interactive_check (בוחן אינטראקטיבי):** שאלת אימות קצרה לעצירת רצף הקריאה.
6. **sequence_end_card (כרטיס גשר / סיום רצף):** מסך חגיגי המופיע בסוף כל שרשור לימודי. מכיל ברכת סיום, שם הנושא שהושלם, כפתור CTA גדול ("המשך לנושא הבא") וכפתור קטן ("חזור להתחלה"). משמש כ-Bridge Card למעבר חלק בין יחידות לימוד.

השדה `type` בכל אלמנט JSON חייב להיות מתוך הרשימה הנ"ל.
בנוסף, כל ריל חייב להכיל את השדה `sequence_title` המגדיר את נושא העל שאליו הוא קשור (כדי שמנגנון הגלילה ייצר כותרת קפואה - Sticky Header - לרצף הרילסים המקושרים).
```

## 4. אסטרטגיית Frontend ופיתוח ממשק משתמש (UI Architecture)  
ארכיטקטורת האתר תתבסס על Component-Based UI המיועד במובהק למובייל (Mobile-First).

* **Scroll Snapping Pattern:** בניית מעטפת התומכת באפקט החלקה. רכיב האב יוגדר עם מחלקות `overflow-y-scroll snap-y snap-mandatory h-screen`. כל רכיב ילד (Reel Card) יוגדר כ-`snap-start h-screen w-full flex flex-col`.  
* **Dumb Components:** קומפוננטות ה-UI לא יכילו מידע קשה (Hardcoded). הן יוגדרו כמקבלות `Props` של אובייקט ה-`bite`.   
* **Dynamic Animation Binding (Registry Pattern):** ה-UI יעשה שימוש ברכיב מתווך (`AnimationFactory`) אשר יקרא את המחרוזת מהשדה `visual_trigger` (למשל `animation_archimedes_vectors`). ה-Factory פועל כ-Registry ויטען באופן דינמי קומפוננטות SVG/Framer Motion ייעודיות מתיקיית `animations/`. במידה והאנימציה טרם פותחה, הוא ירנדר `FallbackAnimation` גנרי.
* **UI Micro-Aesthetics & Collision Avoidance (מניעת התנגשויות ויזואליות):** בכל הוספת אלמנט UI (לדוגמה: Sticky Header, תגית, כפתור או סליידר) על הסוכן לנתח ולחשב את מרחב המחיה של האלמנט. 
  - חובה לוודא שאין חפיפה או הסתרה של טקסטים או תגיות.
  - יש להגדיר שכבות ברורות (`z-index`) וריווח מונע התנגשויות (כמו `pt-24` להרחקה מכותרות קפואות).
  - כל סליידר, כפתור או אלמנט אינטראקטיבי חייב להיות נגיש, בולט ("מגניב"), ולספק פידבק ויזואלי ברור ברמת המיקרו.
* **Loading Animations (Viewport Once):** כל האלמנטים בכל ריל (כותרת, טקסט, אנימציות, רכיבי UI נוספים) מחויבים להיטען עם אנימציית כניסה (`initial` ו-`whileInView` של Framer Motion) *רק בפעם הראשונה* שהמשתמש גולל אליהם. יש להשתמש ב-`viewport={{ once: true, amount: 0.2 }}` על מנת להבטיח שאם המשתמש גולל הלאה וחוזר לאותו ריל, האלמנטים כבר יהיו טעונים ולא יופעלו מחדש.

## 5. מעקב מלאי אנימציות (Animation Triggers Backlog)
קיימים בסך הכל 22 טריגרים ויזואליים שנוצרו במהלך המרת יחידות 1 ו-2 (11 לכל יחידה).
*   **אנימציות ליבה (Core Physics Animations):** פותחו/מפותחות 4 קומפוננטות ראשיות (`animation_archimedes_vectors`, `animation_pascal_law`, `animation_drag_equation`, `animation_metacentric_torque`).
*   **אנימציות חסרות (Technical Debt):** נותרו עוד 18 טריגרים המקבלים כעת עיצוב Fallback (כגון `animation_uhthoff_ms`, `animation_gate_control`, `animation_turbulent_gliding` ועוד). על סוכני התכנון וה-UI לחזור למלאי זה בסשנים עתידיים ולהשלים את פיתוחן הויזואלי.
